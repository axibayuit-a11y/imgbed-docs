# افزودن کانال S3

## چه زمانی مناسب است

از کانال S3 زمانی استفاده کنید که:

- می‌خواهید هر سرویس ذخیره‌سازی اشیای سازگار با S3 را متصل کنید، مانند Cloudflare R2، Backblaze B2، MinIO، AWS S3 یا ارائه‌دهنده سازگار دیگر.
- مدل استاندارد پیکربندی S3 را ترجیح می‌دهید: `Endpoint`، کلید دسترسی و کلید محرمانه.
- نمی‌خواهید از کانال بومی Cloudflare R2 با binding استفاده کنید، یا ارائه‌دهنده ذخیره‌سازی شما Cloudflare R2 نیست.

## پیش از شروع چه چیزهایی لازم است

| مورد | کاربرد |
| --- | --- |
| اندپوینت S3 | اندپوینت API سازگار با S3 برای سرویس ذخیره‌سازی اشیا. |
| نام باکت | باکتی که فایل‌ها در آن ذخیره می‌شوند. |
| ناحیه | ناحیه ذخیره‌سازی. برای Cloudflare R2 معمولاً `auto` است. |
| Access Key ID | شناسه کلید دسترسی S3. |
| Secret Access Key | کلید دسترسی محرمانه S3. |
| دامنه CDN | دامنه سفارشی اختیاری برای دسترسی به فایل‌ها. |

نمونه برای Cloudflare R2 از طریق API سازگار با S3:

```text
Endpoint: https://your-account-id.r2.cloudflarestorage.com
Bucket: your-r2-bucket-name
Region: auto
Access Key ID: copied from the Cloudflare R2 API token
Secret Access Key: copied from the Cloudflare R2 API token
```

## کجا اضافه کنیم

1. تنظیمات سیستم را باز کنید.
2. به تنظیمات بارگذاری بروید.
3. در گوشه بالا سمت راست روی افزودن کانال کلیک کنید.
4. `S3` را انتخاب کنید.

## ساخت کلیدهای API سازگار با S3 برای Cloudflare R2

1. Cloudflare Dashboard را باز کنید.
2. به `R2 Object Storage` بروید.
3. `Manage R2 API Tokens` یا ورودی مدیریت توکن‌های API را پیدا کنید.

![ورودی توکن API در R2](../../image/upload/s3/r2 api token入口.png)

4. یک توکن API برای R2 بسازید که بتواند به باکت مقصد دسترسی داشته باشد.

![ساخت توکن API حساب](../../image/upload/s3/创建账户api令牌.png)

5. مقدارهای تولیدشده `Access Key ID` و `Secret Access Key` را کپی کنید.

![کپی مقدارهای توکن](../../image/upload/s3/复制令牌值.png)

6. به صفحه باکت R2 برگردید و نام باکت را بررسی کنید.
7. اندپوینت API سازگار با S3 حساب را یادداشت کنید. معمولاً شبیه این است:

```text
https://your-account-id.r2.cloudflarestorage.com
```

## مراحل پیکربندی

1. تنظیمات بارگذاری را باز کنید.
2. روی افزودن کانال کلیک کنید.
3. `S3` را انتخاب کنید.
4. یک نام کانال قابل تشخیص وارد کنید، برای مثال `s3test`.
5. اندپوینت API سازگار با S3 را در `Endpoint` وارد کنید.
6. اگر از CDN سفارشی استفاده می‌کنید، آن را در فیلد دامنه CDN سفارشی وارد کنید؛ در غیر این صورت فیلد را خالی بگذارید.
7. نام باکت را وارد کنید.
8. ناحیه را وارد کنید. برای نمونه Cloudflare R2 از `auto` استفاده کنید.
9. Access Key ID و Secret Access Key را وارد کنید.
10. دسترسی path-style را خاموش نگه دارید، مگر اینکه ارائه‌دهنده شما صریحاً آن را لازم بداند.
11. اگر کنترل ظرفیت می‌خواهید، محدودیت سهمیه را فعال کنید و محدودیت ظرفیت و آستانه را وارد کنید.
12. روی ذخیره کلیک کنید.

## روش بررسی

| بررسی | روش بررسی |
| --- | --- |
| کارت کانال نمایش داده می‌شود | پس از ذخیره، صفحه تنظیمات بارگذاری باید کارت کانال S3 را نشان دهد. |
| کانال فعال است | کلید گوشه بالا سمت راست کارت باید روشن بماند. |
| فیلدهای کلیدی ذخیره شده‌اند | نمای جزئیات باید Endpoint، Bucket، Region، دسترسی path-style و فیلدهای مرتبط را نشان دهد. |
| بارگذاری کار می‌کند | یک تصویر آزمایشی بارگذاری کنید و مطمئن شوید شیء در باکت مقصد ایجاد شده است. |
| لینک باز می‌شود | لینک تصویر برگشتی پس از بارگذاری باید به‌صورت عادی باز شود. |
| نمایش ظرفیت کار می‌کند | اگر محدودیت سهمیه فعال باشد، کارت باید ظرفیت مصرف‌شده و محدودیت پیکربندی‌شده را نشان دهد. |

## FAQ

### آیا باید دسترسی path-style را فعال کنم؟

معمولاً خیر.

فقط زمانی آن را فعال کنید که ارائه‌دهنده سازگار با S3 شما از URLهای virtual-hosted-style پشتیبانی نکند، یا مستندات ارائه‌دهنده صریحاً دسترسی path-style را لازم بداند. برخی اندپوینت‌های MinIO، سرویس‌های S3 خودمیزبان و سرویس‌های سازگار با OpenList ممکن است به آن نیاز داشته باشند.

### دامنه CDN چه کاربردی دارد؟

اگر یک CDN یا دامنه دسترسی سفارشی را جلوی باکت قرار داده‌اید، آن را اینجا وارد کنید.

پس از تنظیم، لینک‌های تولیدشده فایل این دامنه را ترجیح می‌دهند. اگر از CDN استفاده نمی‌کنید، فیلد را خالی بگذارید.

### چرا بارگذاری شکست می‌خورد؟

ابتدا این موارد را بررسی کنید:

1. Endpoint شامل URL کامل با `https://` باشد.
2. نام باکت درست باشد.
3. Region با نیاز ارائه‌دهنده مطابقت داشته باشد.
4. Access Key ID و Secret Access Key به‌طور کامل کپی شده باشند.
5. کلید مجوز نوشتن در باکت مقصد را داشته باشد.
6. ارائه‌دهنده به دسترسی path-style نیاز نداشته باشد، یا در صورت نیاز فعال شده باشد.
